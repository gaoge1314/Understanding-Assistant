# 类型驱动升级设计方案

## 概述

基于 v3.txt 中总结的"四类内容类型 × 三层理解路径"框架，对现有理解路径记忆助手进行升级。核心改动：新增诊断 Skill，并对 card-generator（理解模块）和 memory-techniques（技巧模块）进行深度重构，使其根据内容类型切换策略。

## 背景

现有架构对所有知识点使用同一套通用策略（三大机制 + 三种技巧），不做内容类型区分。v3.txt 的分析表明，不同类型的内容（逻辑序列/机制机理/结构模型/概念辨析）需要不同的理解入口和记忆策略。

## 新流水线

```
诊断 Skill（新增） → card-generator（理解，升级）→ memory-techniques（技巧，升级）→ 编排器组装
```

### 与旧流水线的对比

旧：
```
card-generator（通用理解）→ memory-techniques（通用技巧）→ 组装
```

新：
```
type-diagnoser（类型诊断）→ card-generator（类型驱动理解）→ memory-techniques（类型驱动技巧）→ 组装
```

## 模块设计

### 1. 诊断 Skill（新建）

**文件**：`.agents/skills/type-diagnoser/SKILL.md`

**职责**：接收知识点标题 + 教材段落 + 完整知识点清单 → 输出内容类型诊断结果

**输入**：
- 知识点标题
- 对应的教材段落
- 完整知识点清单（用于判断是否与其他概念易混）

**输出格式**：
```json
{
  "content_type": "A | B | C | D | mixed",
  "rationale": "简要理由，说明为什么归为此类",
  "subtype_hints": {
    "has_sequence": true/false,
    "has_causality": true/false,
    "has_comparison": true/false,
    "has_hierarchy": true/false
  }
}
```

**分类规则**：

| 类型 | 特征 | 判断问题 |
|:---|:---|:---|
| A-逻辑序列 | 有先后顺序、因果逻辑 | 它可以分成几个步骤吗？顺序能换吗？ |
| B-机制/机理 | 描述过程运转、动态关系 | 它解释"为什么发生"吗？有因果链吗？ |
| C-结构/模型 | 并列要素组成静态关系 | 它由几个部分构成？要素是并列/层级关系吗？ |
| D-概念辨析 | 易混淆概念需要区分 | 它和另一个概念长得很像吗？容易弄混吗？ |

**规则**：
- `subtype_hints` 至少有 1 个为 true
- 如果多个触发 → `content_type` 设为 `mixed`，保留所有匹配的 hints

**错误处理**：
- 无法判断 → `content_type: "unknown"`，下游使用默认通用策略

### 2. card-generator（理解模块，深度重构）

**文件**：`.agents/skills/card-generator/SKILL.md`

**核心改动**：在现有 Prompt 结构中新增"类型分支入口"，后续所有步骤根据类型切换策略。

#### 2.1 类型分支入口（新增）

在 Prompt 开头（思维中间层检测之前）插入类型分支指令，根据 `content_type` 选择策略组合：

| 类型 | 理解策略 |
|:---|:---|
| A（逻辑序列） | 因果推演 + 分组压缩 + 逻辑轴归纳 |
| B（机制/机理） | 比喻锚定 + 层级图 + 概念对比 |
| C（结构/模型） | 整体意象 + 逐一追问 + 内部接口图 |
| D（概念辨析） | 分叉对比 + 场景代入 + 区分口诀 |
| mixed | 同时启用对应类型的策略组合 |

#### 2.2 decomposition 按类型调整

| 类型 | decomposition 风格 | 典型格式 |
|:---|:---|:---|
| A | 步骤→因果配对 | "步骤1 → 因为..." |
| B | 阶段→比喻配对 | "阶段1 → 就像..." |
| C | 要素→存在理由 | "要素1 → 为什么必须有" |
| D | 分叉点→对比维度 | "共同起点→在哪分叉" |

#### 2.3 scenario_question + judgment_chain 按类型调整

| 类型 | 题目风格 | 判断链风格 |
|:---|:---|:---|
| A | 给定一个打乱顺序的操作序列，要求重新排序并解释原因 | 因果推理链（3-5步，每步追问"为什么"） |
| B | 给定一个具体现象，要求用机制原理解释其发生过程 | 因果推理链（3-5步，侧重因果关系推导） |
| C | 给定一个不完整的结构，要求判断缺少哪个要素并说明影响 | 要素判断链（确认要素→判断关系→得出结论） |
| D | **双场景辨析题**：给出两个相似场景，要求判断分别属于哪个概念 | 分叉判断链（确认共同点→找到分叉点→判断归属） |

**类型D 的 scenario_question 格式要求**：
```
场景A：……（指向概念X）
场景B：……（指向概念Y）
问题：场景A和B分别属于哪个概念？为什么？
```

**类型D 的 judgment_chain 格式要求**：
```
第1步 → 确认共同起点：X和Y都来自……
第2步 → 找到分叉点：区别在于A有……而B没有
第3步 → 匹配场景：场景A满足……所以是X；场景B满足……所以是Y
第4步 → 得出结论：……
```

#### 2.4 三层覆盖检查（★ 新增）

v3.txt 框架要求每张卡片必须覆盖"理论→操作→知识"三个理解层次。生成卡片后，需显式检查三个层次是否都被回答：

| 理解层次 | 对应字段 | 类型A的问题 | 类型B的问题 | 类型C的问题 | 类型D的问题 |
|:---|:---|:---|:---|:---|:---|
| **理论**（为什么） | `core_principle` + decomposition 中的因果解释 | 为什么这个顺序不能乱？ | 这个过程为什么会发生？ | 为什么这几个要素缺一不可？ | 它们为什么不一样？ |
| **操作**（怎么做） | `decomposition` 中的操作/阶段/关系描述 | 每一步具体怎么操作？ | 这个过程分几个阶段？ | 要素之间怎么咬合？ | 怎么快速区分？ |
| **知识**（是什么） | `answer` + 术语在 decomposition 中的标注 | 每一步叫什么名称？ | 每个阶段叫什么？ | 每个要素叫什么？ | 各自的定义是什么？ |

**检查规则**：
- 生成完卡片后，逐个层次检查是否已被覆盖
- 如果某个层次缺失（如类型D 没有解释"各自的定义"），需补充

#### 2.5 不变的部分

- 输出 JSON 字段结构不变（title/answer/core_principle/decomposition/scenario_question/judgment_chain/judgment_conclusion/knowledge_interfaces）
- 思维中间层检测逻辑不变
- 知识接口引用逻辑不变
- **新增**：输出 JSON 中透传 `content_type` 字段（来自诊断结果）

### 3. memory-techniques（技巧模块，升级）

**文件**：`.agents/skills/memory-techniques/SKILL.md`

**核心改动**：输入新增 `content_type`，技巧生成从"通用条件触发"变为"类型导向 + 条件触发"。

#### 3.1 关键词结构重构

旧格式（词+解释混在一起）：
```json
"keywords": ["N（自然灾害输入能量） → 第一推动力"]
```

新格式（拆分）：
```json
"keywords": ["N", "Z", "E", "Na-tech"],
"keyword_explanations": [
    "N：自然灾害输入的能量，是整个过程的第一推动力",
    "Z：载体积累的能量，决定了同等地震下的危险程度",
    "E：耗散的能量，以火灾/爆炸/泄漏三种形式释放",
    "Na-tech：Natural-Technological 的合成词"
]
```

#### 3.2 技巧生成规则（类型导向 + v3.txt 具体策略）

| 类型 | 关键词策略 | 层级图策略 | 对比表策略 | 类型专属技巧 |
|:---|:---|:---|:---|:---|
| A | 侧重步骤关键词 + 顺序标识（"关→堵→倒→注→放"中的核心动词） | 条件触发（流程图风格，强调→方向） | 条件触发 | **强制生成"顺序联想口诀"**：从分解的每组步骤中提取一个字，串成一句话（如"关堵倒注放"→"关赌倒猪放"） |
| B | 侧重核心术语 + 因果标识（提取能触发意象的术语） | **优先生成**（层级图 + 因果箭头标注） | 条件触发（与相似机制对比） | **强制生成"费曼类比"**：在 keyword_explanations 中嵌入一个日常场景的类比（如"一排鞭炮""五种人际关系"） |
| C | 侧重要素名词 + 从属标识（提取所有要素名称） | **优先生成**（树状层级图，根→分支→子要素，2-3层） | 条件触发 | **强制生成"一句话意象"**：在 keyword_explanations 中用一个整体画面串联所有要素（如"灾害系统就是一场悲剧的四要素——舞台、演员、观众、伤亡"） |
| D | 侧重区分维度关键词（提取对比表中的维度词，如"触发源""能量性质"） | 条件触发 | **强制生成**（≥3行，围绕"根本原因→表现差异→后果"展开） | **强制生成"区分口诀"**：在 keyword_explanations 中嵌入一句朗朗上口的口诀（如"并发=一因多果同时发生；串发=因果链环环相扣"） |

#### 3.3 类型专属技巧的输出格式

**类型A 的顺序联想口诀**——嵌入到 `keyword_explanations` 数组中，用特殊前缀标记：
```json
{
  "keyword_explanations": [
    "关：关闭泄漏源（第一步，最安全）",
    "堵：堵塞泄漏点（第二步，仍在源头）",
    "倒：倒罐转移（第三步，开始处理已泄漏的）",
    "【顺序口诀】关堵倒注放 → 关赌倒猪放（谐音联想）"
  ]
}
```

**类型B 的费曼类比**——嵌入到 `keyword_explanations` 数组中：
```json
{
  "keyword_explanations": [
    "N：自然灾害输入能量 → 就像多米诺的第一推",
    "Z：载体积累能量 → 就像紧绷的弓弦，就等外力触发",
    "【类比】整个Na-tech能量转化就像一个被推倒的多米诺骨牌链——自然灾害是第一块牌，工业储罐是第二块牌"
  ]
}
```

**类型C 的一句话意象**——嵌入到 `keyword_explanations` 数组中：
```json
{
  "keyword_explanations": [
    "孕灾环境：舞台——提供背景和能量",
    "致灾因子：演员——释放能量的主角",
    "承灾体：观众——承受后果的一方",
    "灾情：伤亡——悲剧的结果",
    "【一句话意象】灾害系统就是一场悲剧的四要素：舞台、演员、观众、伤亡"
  ]
}
```

**类型D 的区分口诀**——嵌入到 `keyword_explanations` 数组中：
```json
{
  "keyword_explanations": [
    "触发源：自然触发 vs 事故触发（分叉点）",
    "能量性质：自然能+工业能叠加 vs 同质事故能传递",
    "【区分口诀】Na-tech天空来，多米诺地上起",
    "【场景对照】台风洪水→储罐泄漏 = Na-tech；爆炸冲击波→震碎管道 = 多米诺"
  ]
}
```

#### 3.4 每个技巧的生成指令中增加类型提示

示例——类型D的关键词指令增加：
```
类型D（概念辨析）：优先从 judgment_chain 和 comparison_tables 中提取
区分维度的关键词（如"触发源""能量性质"），标注为"辨析锚点"
同时生成"区分口诀"和"场景对照"，嵌入 keyword_explanations
```

### 4. understanding-memory（编排器，轻量改动）

**文件**：`.agents/skills/understanding-memory/SKILL.md`

#### 4.1 流水线改动（第4步"逐知识卡生成"的子流程）

旧：
```
① card-generator(title, text, all_titles, answer) → 卡片JSON
② memory-techniques(卡片JSON + all_titles) → MemoryTechniques
③ 组装
```

新：
```
① type-diagnoser(title, text, all_titles) → 诊断结果
② 用户确认诊断结果（可选） → 用户可修正 content_type
③ card-generator(title, text, all_titles, answer, 诊断结果) → 卡片JSON（含content_type）
④ memory-techniques(卡片JSON + all_titles) → MemoryTechniques（从卡片JSON读取content_type）
⑤ 组装
```

#### 4.2 用户修正诊断结果（★ 新增）

在诊断结果出来后，卡片生成前，给用户确认的机会：

```
问：判断"Na-tech 事件中的能量转化机制"为【类型B（机制/机理）+ 类型D（概念辨析混合）】
是否正确？
A. 正确，继续
B. 修改为其他类型
C. 我不确定，用通用策略
```

用户选择后：
- **A**：继续流水线
- **B**：用户手动指定 `content_type`，覆盖诊断结果
- **C**：`content_type = "unknown"`，下游使用默认通用策略

#### 4.3 输入参数传递

- 调用 `type-diagnoser`：知识点标题、教材段落、完整知识点清单
- 调用 `card-generator`：原有参数 + 新增诊断结果
- 调用 `memory-techniques`：不变（卡片JSON中已嵌入 content_type）

#### 4.4 错误处理新增

| 新问题 | 处理方式 |
|:---|:---|
| 诊断结果为空/invalid | 跳过该知识点，提示"无法判断内容类型" |
| 诊断结果与教材内容明显矛盾 | 以诊断结果为准，不人工干预 |
| 用户手动修正后 | 以用户修正为准，不再调用诊断 |

## 升级影响范围

| 文件 | 操作 | 改动量 |
|:---|:---|:---|
| `.agents/skills/type-diagnoser/SKILL.md` | **新建** | 完整新文件 |
| `.agents/skills/card-generator/SKILL.md` | **深度修改** | Prompt 结构大改，新增类型分支 + 三层覆盖检查 + 类型D场景代入 |
| `.agents/skills/memory-techniques/SKILL.md` | **深度修改** | 关键词结构重构 + 类型导向规则 + 四类专属技巧 |

## 不涉及的范围

- card-generator 的思维中间层检测逻辑不变
- 卡片 JSON 输出字段（除新增 content_type 外）不变
- 输出 Markdown/JSON 的格式不变
- 增量更新逻辑不变
- 其他 Skill（knowledge-extractor, transcript-processor 等）不受影响
