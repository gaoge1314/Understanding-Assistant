# 交互式 HTML 卡包应用设计文档

> 日期：2026-06-03
> 状态：设计稿（待审批）
> 对应版本：第二次更新

## 一、概述

### 1.1 动机

现有知识卡包以 Markdown + JSON 格式输出，是**静态文档**。用户阅读体验局限在"看"的层面，无法进行主动重构、拖拽排序、记录自己的理解。

本次更新将卡包升级为**交互式 HTML 应用**，并新增 `learning-app` Agent 统一管理生成和辅导。

### 1.2 设计目标

- 将静态卡包转为可交互的 HTML 应用
- 用户可记录自己的理解，与AI版本对比（交互重构）
- 用户可拖拽调整卡片顺序、创建分组（排序分组）
- 卡牌四段内容可折叠/展开，状态持久化（折叠面板）
- 用户可在卡片内拖拽重组概念，系统比对答案并反馈（概念重组）
- 模板化架构，避免重复造轮子

### 1.3 本次范围

- **做**：新增 learning-app Agent + HTML 模板 + 生成脚本
- **做**：三栏式布局 + 拖拽排序 + 分组 + 折叠面板 + 用户笔记记录
- **做**：概念拖拽重组（右侧栏核心交互）
- **做**：底部"进入检测"入口 + 右侧栏模式提示标识
- **做**：模板化架构，后续新增科目只需注入数据
- **不做**：背诵模式、记忆曲线、内嵌AI追问、概念填空、移动端适配（记入下次更新规划）

## 二、架构设计

### 2.1 整体架构

```
understanding-memory Skill (现有)
    │
    ├── 生成卡包 JSON → output/json/{subject}_v{version}.json
    │
    ▼
learning-app Agent (★ 新增)
    │
    ├── ① 读取 JSON 卡包数据
    ├── ② 拷贝 templates/ 中的 HTML/CSS/JS 模板
    ├── ③ 卡包数据注入 → data.json
    └── ④ 输出至 output/card-app/
         ├── index.html    (模板，不动)
         ├── style.css     (模板，不动)
         ├── app.js        (模板，不动)
         └── data.json     (★ 唯一变化部分)
```

### 2.2 模板化架构

核心思路：HTML/CSS/JS 作为通用模板存放在 Agent 目录下，每次只需替换 data.json。

```
.agents/skills/learning-app/
├── SKILL.md                    # Agent prompt
├── templates/                  # ★ 可复用模板文件
│   ├── index.html              # 页面骨架（无需修改）
│   ├── style.css               # 样式（无需修改）
│   └── app.js                  # 交互逻辑（无需修改）
└── scripts/
    └── generate_app.py         # 读取JSON → 拷贝模板 → 注入数据 → 输出
```

### 2.3 新增 Agent：learning-app

**触发方式**：用户说"生成学习应用" / "打开交互式卡包" / 或其他等价表述。

**处理流程**：

```
Step 1: 确认已有卡包 JSON
        → 检查 output/json/ 下的最新版本
        → 如无卡包，引导用户先使用 understanding-memory 生成

Step 2: 执行 generate_app.py
        → 读取 JSON 数据
        → 对每张卡片动态判断 reconstruct_type（规则见 3.2 功能4）
        → 将 JSON 转换为 data.json 注入（含 reconstruct_type 字段）
        → 拷贝 templates/ 到 output/card-app/

Step 3: 输出完成通知
        → 告知用户 output/card-app/index.html 可直接打开

Step 4: AI 辅导模式（对话式）
        → 用户带着问题回到 Trae 对话
        → Agent 读取当前学习状态，提供引导
```

**输入**：`output/json/{subject}_v{version}.json`
**输出**：`output/card-app/` 目录（可直接双击 index.html 打开）

## 三、HTML 应用设计

### 3.1 布局

```
┌──────────────────────────────────────────────────────────┐
│  📚 灾害学知识卡包            [交互模式 🔄 背诵模式]  ⚙️  │  ← 顶部导航
├──────────┬────────────────────────────┬──────────────────┤
│ 左侧边栏  │       主内容区              │    右侧边栏       │
│ (目录)   │      (当前卡片)             │  (重构交互区)     │
│          │                            │                  │
│  ▸ 分组A │  💡 我的理解 ──────────    │  模式: 🔄 概念重组│
│    ✓ 卡1 │  (用户笔记展示区)           │  ┌────────────┐ │
│      卡2 │  ────────────────────      │  │ 拖拽目标区域 │ │
│      卡3 │  ① 答案 (折叠可展开)        │  │ 位置1: ___ │ │
│  ▸ 分组B │  ② 理解路径 (折叠可展开)     │  │ 位置2: ___ │ │
│      卡4 │  ③ 记忆技巧 (折叠可展开)     │  └────────────┘ │
│      卡5 │  ④ 知识接口 (折叠可展开)     │  可拖拽:        │
│          │                            │  [A] [B] [C]   │
│  + 新建… │                            │  ────────────  │
│          │                            │  我的理解: [__] │
├──────────┴────────────────────────────┴──────────────────┤
│   [📝 进入检测 · 下一版支持]                               │  ← 底部
└──────────────────────────────────────────────────────────┘
```

### 3.2 四大功能详细设计

#### 功能1：折叠面板

- 每张卡片的四段内容（答案、理解路径、记忆技巧、知识接口）均为折叠面板
- 默认展开"答案"和"理解路径"，折叠"记忆技巧"和"知识接口"
- 用户可点击标题手动展开/折叠
- 折叠状态保存到 localStorage，下次打开保持

**技术实现**：使用 `<details>` HTML 元素 + 自定义 CSS。

#### 功能2：拖拽排序 + 分组

- 左侧边栏目录支持拖拽调整卡片顺序
- 用户可创建分组，将卡片拖入分组
- 分组可折叠/展开，名称可编辑
- 排序和分组状态保存到 localStorage
- 提供"恢复默认顺序"按钮

**技术实现**：HTML5 Drag & Drop API + localStorage 持久化。

**数据结构**：

```javascript
{
  cardOrder: [3, 1, 2, 4],        // 卡片顺序（索引列表）
  groups: [                        // 用户创建的分组
    { name: "基础概念", cardIds: [0, 1] },
    { name: "灾害管理", cardIds: [2, 3] }
  ]
}
```

#### 功能3：用户笔记

**布局**：输入与展示分离

| 区域 | 位置 | 说明 |
|:---|:---|:---|
| **笔记展示** | 主内容区，卡片内容顶部 | 始终可见，含"我的理解"标题+用户文本+清除按钮 |
| **笔记输入（有概念重组）** | 右侧栏，拖拽区底部 | 浓缩为单行输入框+提交按钮 |
| **笔记输入（无概念重组）** | 右侧栏，占满全区域 | 扩展为多行输入框 |

**交互**：
- 用户在任何位置的输入框中写完后点击提交，内容保存到 localStorage
- 笔记展示区实时更新
- 支持清除笔记
- 再次打开时笔记仍在，可与 AI 版本并列对比

**数据结构**：

```javascript
{
  cardStates: {
    "灾害系统四要素": {
      folded: { answer: true, path: true, memory: false, interface: false },
      myUnderstanding: "灾害就是系统出问题了...",
      lastReviewed: "2026-06-03"
    }
  }
}
```

#### 功能4：概念拖拽重组 ★

针对概念结构类卡片（如"灾害系统四要素"、"灾害管理7阶段"），右侧栏提供拖拽排序交互训练。

> 注意：无论是否启用概念重组，右侧栏底部始终保留"我的理解"输入框（单行浓缩版）。

**交互流程**：

```
① 系统识别当前卡片类型（概念结构类）
② 从卡片数据中提取关键概念块（如四要素的名称）
③ 打乱顺序，以可拖拽块的形式展示在右侧栏
④ 用户将概念块拖到正确的逻辑位置（四象限/时间轴/因果链）
⑤ 点击"提交比对"
⑥ 系统比对用户排序 vs 正确答案，逐项指出错误并附解释
```

**界面示意**：

```
┌─ 重构交互 ──────────────┐
│ 将概念拖到正确位置：      │
│                          │
│  ┌──────────────────┐   │
│  │  拖拽目标区域      │   │
│  │  位置1: _____     │   │
│  │  位置2: _____     │   │
│  │  位置3: _____     │   │
│  │  位置4: _____     │   │
│  └──────────────────┘   │
│                          │
│  可拖拽概念块：           │
│  [致灾因子] [承灾体]      │
│  [孕灾环境] [灾情]       │
│                          │
│  [提交比对]              │
│                          │
│  比对结果：               │
│  ✓ 孕灾环境 → 位置1      │
│  ✗ 致灾因子 → 位置2      │
│     → 解释：...          │
└──────────────────────────┘
```

**比对逻辑**：
- 卡片 JSON 中可通过 `decomposition` 的顺序推断正确位置
- 每个放错位置的概念块，系统显示"为什么这个应该放这里"的解释（从 `core_principle` 或 `judgment_chain` 中提取）
- 全部正确时显示"✅ 全部正确！"并自动标记该卡片 `isCompleted: true`

**触发条件**：`reconstruct_type` 字段值为 `"reorder"` 时自动启用。该字段由 learning-app Agent 在生成 data.json 时动态判断（不修改原始卡包 JSON），规则如下：

| 规则 | 说明 |
|:---|:---|
| decomposition 条目数 ≥ 2 | 至少2个概念块才有重组意义 |
| 条目为平行关系 | 各条目是可互换顺序的平行概念，而非因果链（后一步依赖前一步） |
| 标题含指示词 | 知识点标题含"要素"、"类型"、"分类"、"阶段"、"步骤"、"组成部分"等 |

示例：
- ✅ `"灾害系统四要素"` → `reconstruct_type: "reorder"`（四要素是平行概念）
- ❌ `"灾害系统动力学机制"` → `reconstruct_type: null`（分解是因果推演）

### 3.3 localStorage 完整数据结构

```javascript
  lastUpdated: "2026-06-03",
  cardOrder: [0, 1, 2, 3],            // 排序后的卡片索引

```

data.json 直接继承卡包 JSON 的结构，保持兼容：

```json

  "source_file": "灾害学复习资料.docx",
  "generated_at": "2026-06-03T10:00:00",
  "version": 1,
  "all_knowledge_titles": ["灾害系统四要素", "灾害系统动力学机制", "..."],
  "cards": [
    {
      "title": "灾害系统四要素",
      "answer": "灾害系统由孕灾环境、致灾因子、承灾体和灾情四要素构成",
      "core_principle": "灾害不是一件事，而是一个失效的系统",
      "decomposition": ["..."],

      "knowledge_interfaces": ["..."]
    }
  ]
}
```

### 3.5 右侧栏模式标识与 AI 辅导入口

- 右侧栏顶部显示当前卡片的交互模式标识：

| 标识 | 卡片类型 | 右侧栏交互 |
|:---|:---|:---|
| 🔄 概念重组 | 概念结构类 | 拖拽概念块到正确位置 |
| 📝 笔记模式 | 其他类型 | 退化为"我的理解"输入框 |

- 右侧栏底部有一个"在 Trae 中提问"按钮
- 点击后提示用户回到 Trae 对话，并告知 Agent 当前学习的卡片标题
- 用户回到 Trae 后，Agent 主动询问当前学习状态，用户手动告知
- Agent 据此提供 AI 辅导和引导

### 3.6 底部检测入口

- 页面底部有一个"进入检测"按钮（本次为占位，展示入口）
- 鼠标悬停显示提示："下一版将支持检测题和批改反馈"
- 点击后提示功能开发中

## 六、关键技术决策

### 6.1 为什么用纯前端（无框架）

- 零依赖：无需 Node.js / npm / 构建工具
- 零部署：双击 index.html 即可运行（file:// 协议）
- 零维护：模板写好后再不动，所有逻辑在 app.js 中

### 6.2 为什么用 localStorage 而非文件

- 用户的学习状态（笔记、排序、折叠）应在浏览器中持久化
- 本地文件读写需 Node.js 环境，违背纯前端原则
- localStorage 足够存储所有用户数据（按 KB 计，远未达 5MB 限制）

### 6.3 为什么 Agent 不内嵌到 HTML 中

- 本次目标：先让 HTML 应用独立可用
- 内嵌 AI 通信需要 Agent 后台监听文件变化，复杂度高且依赖 Trae 运行环境
- 保留扩展性：后续可在 app.js 中增加文件通信接口

## 七、设计原则检查

| 原则 | 状态 | 说明 |
|:---|:---|:---|
| YAGNI | ✅ | 只做本次范围的4个功能，不做背诵模式和记忆曲线 |
| 模板化 | ✅ | 模板一次构建永久复用，新增科目只改 data.json |
| 纯前端 | ✅ | 零依赖，双击可用 |
| 离线可用 | ✅ | 所有资源自包含，无外部 CDN 引用 |
| 用户控制 | ✅ | 用户数据存在本地，完全自主管理 |
