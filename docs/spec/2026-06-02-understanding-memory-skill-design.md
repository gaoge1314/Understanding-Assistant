## 一、概述

### 1.1 核心理念

从"技巧式记忆术"升级为"理解型记忆"。目标不是帮学生记住孤立的知识点，而是帮他们构建**知识图谱**，让知识点互为锚点，最终能**用知识做复杂判断**。

### 1.2 理解的四个层次

| 层次 | 定义 |
|:---|:---|
| **联系** | 把新知识挂到旧知识的钩子上 |
| **解释** | 能用自己的话讲清楚"为什么" |
| **迁移** | 在看似无关的领域认出同一个原理 |
| **生成** | 能从理解中推导出新东西 |

### 1.3 第一阶段范围

- **做**：用户上传文件 → AI 生成"理解路径卡包"（Markdown + JSON 双格式）
- **不做**：交互式检测问答、判析反馈、用户回答批改（留到后续阶段）

### 2.2 两阶段生成策略（解决知识接口悬空问题）

| 阶段 | 动作 | 产出 | 持久化 |
|:---|:---|:---|:---|
| **第一遍** | 扫描全部内容，识别所有知识点 | `all_knowledge_titles` 完整知识点清单 | ✅ 保存到文件 |
| **第二遍** | 逐知识点生成卡片，引用完整清单 | 每张卡的"知识接口"字段 | ✅ 卡包 JSON |

**增量更新流程**（后续上传新材料时）：
1. 读取已保存的 `all_knowledge_titles` 清单
2. 从新材料中提取新知识点，追加到清单
3. 生成新卡片（知识接口可引用全部新旧知识点）
4. 扫描已有旧卡片的"知识接口"字段，如有与新知识点的关联 → 自动追加链接
5. 保存更新后的清单和卡包

### 2.3 组件职责

| 组件 | 职责 | 技术选型 |
|:---|:---|:---|
| **parser.py** | PDF/Word/TXT → 纯文本 | pdfplumber, python-docx |
| **models.py** | 数据结构定义 | Python dataclass |
| **knowledge_store.py** | 知识点清单读写、卡包版本管理 | Python JSON |
| **card_generator.py** | Markdown/JSON 格式生成 | Python 标准库 + json |
| **skill.md** | AI 指令 + 质量控制 + 思维补全 | Trae AI Skill 格式 |

### 3.2 数据结构 (models.py)

```python
@dataclass
class KnowledgeCard:
    title: str                          # 【知识点】名称
    core_principle: str                 # 【核心原理】
    problem_solved: str                 # 【它解决了什么问题】
    decomposition: List[str]            # 【分解理解】（每步有"为什么"）
    knowledge_interfaces: List[str]     # 【与其他知识的关键接口】
    scenario_question: str              # 【典型判断情境】题目
    scenario_answer: str                # 【典型判断情境】结论
    judgment_chain: List[str]           # ★ 【判断链】推理步骤

@dataclass
class CardPack:
    subject: str
    source_file: str
    cards: List[KnowledgeCard]
    generated_at: str
    version: int = 1                          # ★ 版本号（增量更新时+1）
    all_knowledge_titles: List[str] = None    # ★ 完整知识点清单（两阶段产物）
```

### 3.3 knowledge_store.py（新增模块）

**职责**：管理持久化的中间产物

| 文件 | 内容 | 用途 |
|:---|:---|:---|
| `output/knowledge/{subject}_titles.json` | 知识点标题清单 | 第一次扫描后保存，供增量更新 |
| `output/knowledge/{subject}_v{version}.json` | 卡包完整数据 | 每次生成/更新后保存 |

**增量更新逻辑**：
```
def update_card_pack(existing_pack, new_text):
    # 1. 从 new_text 提取新知识点
    new_titles = extract_knowledge_titles(new_text)
    # 2. 合并到完整清单
    all_titles = existing_pack.all_knowledge_titles + new_titles
    # 3. 生成新卡片
    new_cards = generate_cards(new_text, all_titles)
    # 4. 扫描旧卡片，补充新链接
    for card in existing_pack.cards:
        new_links = find_new_connections(card, new_titles)
        card.knowledge_interfaces.extend(new_links)
    # 5. 合并输出
    existing_pack.cards.extend(new_cards)
    existing_pack.all_knowledge_titles = all_titles
    existing_pack.version += 1
```

| 文件类型 | 库 | 策略 |

| `.docx` | `python-docx` | 提取段落，保留标题层级 |

## 四、Skill Prompt 设计

### 4.1 处理流程

```
第一遍扫描：
  └→ 通读全文 → 识别所有独立知识点 → 输出 all_knowledge_titles 清单

第二遍生成（对每个知识点）：
  └→ ① 思维中间层检测与补全（新）
  └→ ② 三大催化机制应用
  └→ ③ 按卡片模板填充内容
  └→ ④ 知识接口引用 all_knowledge_titles 中的条目
```

### 4.2 卡片模板（强制字段）

每张卡片包含以下字段，顺序固定：

```
【知识点】名称
  ├─ 🔍 典型判断情境
  │   ├─ 题目（全新情境）
  │   ├─ 判断链（★ 推理步骤，非结论）
  │   └─ 结论
  ├─ 💡 核心原理（一句话 ≤30 字）
  ├─ 🎯 它解决了什么问题
  ├─ 🔧 分解理解（每步挂"为什么"）
  └─ 🔗 与其他知识的关键接口（≥2 个，可引用完整清单）
```

### 4.3 四大处理机制

#### 机制一：思维中间层检测与补全（★ 新增）

- **触发条件**：检测到课本直接从 A 跳到 C，省略了中间的 B
- **规则**：
  | 缺失类型 | 表现 | AI 补全动作 |
  |:---|:---|:---|
  | **缺背景知识** | 直接使用未定义过的术语/概念 | 插入"前置概念速览"：*"要懂这个，你只需要先知道……"* |
  | **缺逻辑跳跃** | 结论来得突兀，缺少推导步骤 | 插入"逻辑桥梁"：*"为什么A能推出C？因为中间还藏着一个B：……"* |
  | **缺感官体验** | 概念过于抽象，无法直观感受 | 创建"思想实验"或"极端比喻"：*"想象你是一个水分子，从根部到叶脉，你经历了……"* |

- **执行时机**：在三大机制应用之前，对每个知识点先做"缺省检测"

#### 机制二：原理→操作演绎链

- **适用场景**：知识点涉及操作步骤、流程、方法
- **规则**：不罗列步骤。每步都解释"这一步要解决什么问题 → 原理上为什么这样能解决"
- **反例**："步骤1.胸外按压30次 步骤2.人工呼吸2次"
- **正例**："胸外按压30次 → 为什么？心脏骤停时血液中还有氧，首要任务是让血液流动起来"

#### 机制三：分层抽象塔

- **适用场景**：知识点有多层结构，或包含目的-手段关系
- **规则**：按"目的层 → 策略层 → 具体知识层"三层组织
- **示例**：秦朝郡县制 → 目的层（如何管理庞大国家）→ 策略层（中央垂直管理）→ 具体知识层（长官任免制）

#### 机制四：差异与边界地图

- **适用场景**：遇到相似概念、易混淆术语
- **规则**：用表格对比，核心列是"差异的根本原因"、"该原因如何必然导致表现差异"、"该原因如何必然决定应用场景"
- **示例**：Na-tech 事件 vs 多米诺效应（触发源不同→驱动方式不同→适用范围不同）

### 4.4 理解的4层次 → 卡片字段对应关系

| 理解层次 | 对应卡片字段 | 落地规则 |
|:---|:---|:---|
| **联系**（挂到旧知识） | 🔗 与其他知识的关键接口 | 每张卡至少链接2个已有知识点，说明关系性质 |
| **解释**（说清为什么） | 🔧 分解理解 | 每步操作/概念都必须有"为什么"解释 |
| **迁移**（新场景判断） | 🔍 典型判断情境·题目 | 题目必须是一个全新情境，不能照搬课本例子 |
| **生成**（推导新结论） | 🔍 典型判断情境·判断链 | 答案必须展示完整的推理步骤，而非只给结论 |

### 4.5 质量控制规则

1. 每张卡必须有"典型判断情境"字段（含题目+判断链+结论），不得省略
2. "分解理解"的每一步都必须有"为什么"解释
3. 同一卡包内，卡片之间通过"知识接口"字段形成网络，不得孤立
4. 四大机制需被覆盖，但不为了用机制而生硬套用
5. Markdown 格式美观可读，JSON 格式严格符合 schema

## 九、预处理模块（v3 新增）

### 9.1 动机

用户提供的材料不限于教材，还包括课堂录音文字版、课堂笔记等非结构化文本。这些材料需要先经过预处理，才能用于知识点提取和卡片生成。

### 9.2 新增组件

| 组件 | 类型 | 职责 |
|:---|:---|:---|
| **transcript-processor** | 子 Skill | 录音文字版/笔记的清洗、主题拆分、按主题重组 |
| **output/processed/** | 目录 | 保存预处理中间产物（`{原始名称}_已处理.md`、`{主题}_已整合.md`） |

### 9.3 处理流程

```
用户输入（课堂录音文字版/笔记/教材）
        │
        ▼
┌──────────────────────────────┐
│  transcript-processor        │ ← 新增子 Skill
│                              │
│  ① 类型识别                  │
│    ├─ 口语化、时间顺序 → 录音稿│
│    ├─ 碎片化、简写 → 课堂笔记 │
│    └─ 结构完整 → 教材（跳过） │
│                              │
│  ② 逐文档独立处理             │
│    ├─ 清洗（去除填充词、合并   │
│    │  重复、保留原话「」、纠错）│
│    ├─ 主题拆分（识别话题转换点）│
│    └─ 重组（时间→主题顺序）    │
│    输出：{原始名称}_已处理.md  │
│                              │
│  ③ 跨文档统一整合             │
│    ├─ 合并重叠内容             │
│    ├─ 教材内容为骨架           │
│    ├─ 原话为补充              │
│    └─ 排序（教材章节/逻辑序）  │
│    输出：{主题}_已整合.md      │
└──────────┬───────────────────┘
           │
           ▼
     进入原有流程：识别知识点 → 生成卡片 → 输出卡包
```

### 9.4 对主 Skill（understanding-memory）的影响

只修改第二步"文件预处理"，增加分支：

| 新增文件类型 | 操作 |
|:---|:---|
| 课堂录音文字版 | 调用 **transcript-processor** → `{原始名称}_已处理.md` |
| 课堂笔记 | 调用 **transcript-processor** → `{原始名称}_已处理.md` |

多个文档预处理后，再调用整合功能输出 `{主题}_已整合.md`，下游流程不变。
