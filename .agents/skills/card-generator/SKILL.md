# 类型驱动卡片生成器

**你只负责一个任务：为一个知识点生成一张完整的"理解路径"卡片。**

**关键升级：卡片生成策略由内容类型驱动。** 下方 `┌─ 类型分支 ─┐` 块决定了本卡片的生成策略。每次调用前必须先判断内容类型。

## 输入

调用方会提供：

1. **知识点标题**：当前要处理的卡片主题
2. **对应的教材段落**：从原文中提取的该知识点相关文本
3. **完整知识点清单**：本课程/教材中所有知识点的标题列表（用于知识接口引用）
4. **答案（可选）**：用户指定或 AI 提取的答案文本
5. **内容类型诊断结果**（来自 type-diagnoser）：
   - `content_type`：`A | B | C | D | mixed | unknown`
   - `subtype_hints`：`{ has_sequence, has_causality, has_comparison, has_hierarchy }`

---

## 类型分支指令（放在处理流程之前）

根据 `content_type` 决定整张卡片的生成策略：

| 类型 | 核心策略 | decomposition 风格 | judgment_chain 风格 | scenario_question 风格 |
|:---|:---|:---|:---|:---|
| **A 逻辑序列** | 因果推演 + 分组压缩 + 逻辑轴归纳 | 步骤→原因配对，每个步骤追问"为什么先做这个" | 因果推理链（顺序判断：哪步在前→为什么→后果） | 新场景中按正确顺序排列步骤 |
| **B 机制/机理** | 比喻锚定 + 层级图 + 概念对比 | 阶段→比喻配对，用熟悉模型嫁接抽象关系 | 因果推理链（能量/信息传递：原因→传递→阈值→结果） | 新场景中追踪因果路径 |
| **C 结构/模型** | 整体意象 + 逐一追问 + 内部接口图 | 要素→追问存在理由，逐一回答"为什么必不可少" | 要素判断链（确认要素→判断关系→得出结论） | 新场景中判断要素是否齐全 |
| **D 概念辨析** | 分叉对比 + 场景代入 + 区分口诀 | 分叉点→对比维度配对，标出"在哪一步开始不同" | 分叉判断链（找共同起点→找分叉点→判断归属） | **双场景题**：给出两个相似场景，要求判断分别属于哪个概念 |
| **mixed** | 同时启用对应类型策略的组合 | 复合风格（主类型优先） | 复合风格 | 复合风格 |
| **unknown** | 回退到通用策略（原版三大机制） | 通用风格 | 通用风格 | 通用风格

## 处理流程

对每个知识点，按顺序执行以下步骤：

### 步骤1：思维中间层检测与补全

课本经常从 A 直接跳到 C，省略中间的 B。检测并补全这些缺失的 B。

| 检测类型 | 判断方法 | 补全动作 |
|:---|:---|:---|
| 缺背景知识 | 知识点的前置概念是否默认读者已知但教材未说明 | 插入"前置概念速览"：**要懂这个，你只需要先知道：……** |
| 缺逻辑跳跃 | 原理到结论之间缺少中间推导步骤 | 插入"逻辑桥梁"：**为什么A能推出C？因为中间还藏着一个被忽略的B：……** |
| 缺感官体验 | 概念过于抽象，纯文字描述难以形成直觉 | 创建"思想实验"或"极端比喻"：**想象你是一个X，你经历了……** |

将补全的内容作为当前知识点的元信息记录，**不单独作为卡片字段输出**，而是融入到下面的卡片字段中（尤其是"分解理解"和"核心原理"）。

### 步骤2：卡片内容构建

理解的四层次和卡片字段的对应关系：

| 理解层次 | 对应卡片字段 | 要求 |
|:---|:---|:---|
| ① 联系 | 🔗 与其他知识的关键接口 | 每张卡至少链接 2 个已有知识点，说明关系性质 |
| ② 解释 | 🔧 分解理解 | 按类型风格：A因果配对 / B比喻配对 / C要素追问 / D分叉对比 |
| ③ 迁移 | 🔍 典型判断情境（题目） | 按类型风格：A顺序排序 / B因果追踪 / C要素判断 / D双场景对比 |
| ④ 生成 | 🔍 判断链 + 结论 | 按类型风格：A因果推理 / B传递链 / C要素链 / D分叉链 |

### 步骤3：按内容类型填写 decomposition

根据类型分支指令中的策略表，按类型风格填写 `decomposition` 字段：

- **类型A**：每对"步骤→原因"，强调顺序不可逆的因果约束。如果步骤>4，先**分组压缩**为 2-3 组，每组内部再展开
- **类型B**：每对"阶段→比喻"，用熟悉模型嫁接。第一项应为整体比喻/类比，后续展开阶段
- **类型C**：每对"要素→存在理由"，逐一回答"为什么必不可少"。额外说明要素之间的内部接口（如能量流动、数据传递）
- **类型D**：每对"分叉点→对比维度"，先标出共同起点，再标出分叉点
- **mixed**：主类型优先，辅以次要类型的风格补充
- **unknown**：回退通用策略

### 步骤4：应用关键字高亮

在填写卡片的所有文字字段时，使用统一的高亮格式：

- **粗体**：关键术语、核心概念（第一次出现时使用）
- `行内代码`：公式、数值、专有名词、缩写
- *斜体*：需要强调但非术语的内容（谨慎使用）
- → 箭头：表示因果关系或流程方向

### 步骤5：可选嵌入 Flowchart TB 层级图

如果知识点有明显的层次结构（如分类体系、组成部分），在 `decomposition` 字段末尾附加一段 Mermaid `flowchart TB` 代码，用可视化展示其内部结构。

格式要求：
- 只有层次结构明显时才需要，不要强求
- 将 Mermaid 代码作为 `decomposition` 的最后一个元素
- 在代码块前加一行 `<!-- mermaid -->` 标记

示例：
```json
{
  "decomposition": [
    "步骤1 → 解释……",
    "步骤2 → 解释……",
    "<!-- mermaid -->\n```mermaid\nflowchart TB\n    A[灾害系统] --> B[孕灾环境]\n    A --> C[致灾因子]\n    A --> D[承灾体]\n    A --> E[灾情]\n```"
  ]
}
```

### 步骤6：编写判断链（按类型格式化）

典型判断情境的答案部分，**必须展示完整判断链**，而非仅给结论。

根据内容类型采用不同的判断链格式：

- **类型A**：因果推理链格式
  ```
  第1步 → 在场景中识别……状态（确认起点）
  第2步 → 判断第1步应该做什么（为什么是这一步而不是其他）
  第3步 → 第1步完成后，自然过渡到第2步（为什么）
  第4步 → 重复直到结束
  第5步 → 最终状态/结论
  ```

- **类型B**：传递因果链格式
  ```
  第1步 → 识别初始触发条件（能量/信息输入）
  第2步 → 追踪传递路径（怎么传递的？）
  第3步 → 判断是否超过阈值（临界条件）
  第4步 → 最终结果/状态
  ```

- **类型C**：要素判断链格式
  ```
  第1步 → 确认要素A是否存在（判断标准）
  第2步 → 确认要素B是否存在
  第3步 → 判断要素之间的关系（内部接口）
  第4步 → 得出结论：要素完备/缺失
  ```

- **类型D**：分叉判断链格式
  ```
  第1步 → 确认共同起点：A和B都满足……
  第2步 → 找分叉点：在XX这一点上，A和B开始不同
  第3步 → 按分叉点判断当前场景归属哪个概念
  第4步 → 得出结论：属于X，因为Y
  ```

- **mixed**：主类型格式优先，辅以次要类型补充
- **unknown**：通用判断链格式

### 步骤7：三层覆盖检查（新增）

完成所有字段后，逐层验证本卡片是否覆盖了"理论→操作→知识"三层理解：

| 层次 | 对应问题 | 覆盖检查 | 示例（各类型不同） |
|:---|:---|:---|:---|
| **理论层**（为什么） | 追问因果/原理 | 检查 `core_principle` 是否回答了"为什么" | A: 为什么这个顺序不能乱？ D: 它们为什么不一样？ |
| **操作层**（怎么做） | 建立过程/步骤 | 检查 `decomposition` 是否描述了"怎么做" | B: 这个过程分几个阶段？ C: 要素之间怎么咬合？ |
| **知识层**（是什么） | 填充术语/定义 | 检查 `answer` 是否包含了关键术语的定义 | 所有类型：每个概念/步骤定义了名称 |

**验证规则**：
1. 三层分别对应 `core_principle`（理论）、`decomposition`（操作）、`answer`（知识）
2. 如果某一层缺失或不充分，补充对应的内容
3. 补充的内容必须符合当前内容类型的风格

## 输出格式

严格输出 JSON 格式，不要加注释或额外文字。
注意：本 Skill 不输出 `memory_techniques` 字段，该字段由 `memory-techniques` 子 Skill 单独生成。

```json
{
  "title": "知识点名称",

  "content_type": "A | B | C | D | mixed | unknown（从诊断结果透传）",

  "answer": "考试默写内容（如果调用方提供则直接使用，否则自行生成）",
  "answer_source": "user_specified | ai_extracted | ai_generated",

  "core_principle": "一句话讲透本质（≤30字）",
  "problem_solved": "它解决了什么问题（目的导向）",
  "decomposition": [
    "步骤/概念1 → 因为/为什么……（按类型风格）",
    "步骤/概念2 → 因为/为什么……"
  ],
  "scenario_question": "给出一道全新情境的题目（类型D应为双场景对比题）",
  "judgment_chain": [
    "第1步 → 确认……（按类型格式）",
    "第2步 → 推理……",
    "第3步 → 得出结论……"
  ],
  "judgment_conclusion": "最终结论（原 scenario_answer）",

  "knowledge_interfaces": [
    "→ 知识点X：关系说明（如'我是X的前提'）",
    "→ 知识点Y：关系说明"
  ]
}
```
